What is Kairo

Kairo is a 2D game engine written from scratch in C++17. It uses OpenGL 4.6 for rendering and implements all major engine systems without relying on off-the-shelf libraries for its core architecture.

The engine includes:

  • An archetype-based Entity Component System (ECS) with compile-time queries
  • A batch renderer with texture arrays, layer sorting, and z-depth ordering
  • 2D physics with AABB and circle colliders, spatial grid broadphase, raycasting, and joints
  • 2D lighting with point lights, shadows, and post-processing (bloom, vignette)
  • Lua scripting with hot-reload support
  • An ImGui-based editor with entity hierarchy, inspector, profiler, and gizmos
  • Audio via miniaudio with spatial sound support
  • Gameplay utilities: tilemap, A* pathfinding, AI steering, animation state machines, timers, tweens
  • Custom memory allocators: pool, linear, and stack
  • A job system for multithreaded workloads

The engine is roughly 11,000 lines of C++ and comes with a demo top-down arena shooter that exercises most systems end-to-end.

Building from Source

Requirements

  • C++17 compiler (GCC 9+ or Clang 10+)
  • CMake 3.20+
  • GLFW 3.3+ (install via your package manager)

Everything else is bundled in external/: GLAD, stb_image, ImGui, Lua 5.4, miniaudio, nlohmann/json.

Build Steps

// Install GLFW (Ubuntu/Debian)
// Run in terminal:
// sudo apt install libglfw3-dev

// Build
// cmake -B build -DCMAKE_BUILD_TYPE=Release
// cmake --build build -j$(nproc)

// Run the demo
// ./build/bin/sandbox

// Run tests (263 of them)
// ./build/bin/tests

Or copy-paste these commands directly:

# install GLFW (Ubuntu/Debian)
sudo apt install libglfw3-dev

# build
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)

# run the demo
./build/bin/sandbox

# run tests (263 of them)
./build/bin/tests

Developed on Linux. Should be reasonably portable to other platforms with OpenGL 4.6 support, but Windows and macOS builds have not been tested.

Your First Game

Here is a minimal Kairo application that creates a white square and moves it across the screen. Let's walk through each part.

The Full Example

#include "core/engine.h"
#include "core/application.h"
#include "graphics/renderer.h"
#include "graphics/camera.h"
#include "ecs/world.h"

struct Transform { kairo::Vec3 position; float rotation = 0.0f; };
struct Sprite    { kairo::Vec4 color; kairo::Vec2 size; };

class MyGame : public kairo::Application {
    kairo::World m_world;
    kairo::Camera m_camera;

    void on_init() override {
        m_camera.set_orthographic(1280.0f, 720.0f);
        auto e = m_world.create();
        m_world.add_component(e, Transform{{ 0, 0, 0 }});
        m_world.add_component(e, Sprite{{ 1, 1, 1, 1 }, { 32, 32 }});
    }
    void on_update(float dt) override {
        m_world.query<Transform, Sprite>([&](kairo::Entity, Transform& t, Sprite&) {
            t.position.x += 50.0f * dt;
        });
    }
    void on_render() override {
        kairo::Renderer::begin(m_camera);
        m_world.query<Transform, Sprite>([](kairo::Entity, Transform& t, Sprite& s) {
            kairo::Renderer::draw_quad(t.position, s.size, t.rotation, s.color);
        });
        kairo::Renderer::end();
    }
};

int main() {
    kairo::Engine engine;
    if (!engine.init({ { "My Game", 1280, 720, true } })) return 1;
    MyGame game;
    engine.run(game);
    engine.shutdown();
}

Step-by-Step Breakdown

1. Define Components

Components are plain C++ structs. They hold data only -- no behavior. Here we define Transform (position and rotation) and Sprite (color and size).

struct Transform { kairo::Vec3 position; float rotation = 0.0f; };
struct Sprite    { kairo::Vec4 color; kairo::Vec2 size; };

2. Create Your Application

Subclass kairo::Application and override the lifecycle methods. The engine calls these for you in the right order.

  • on_init() -- runs once at startup. Create your entities here.
  • on_update(float dt) -- runs every frame with the time since the last frame. Put gameplay logic here.
  • on_render() -- runs every frame after update. Draw everything here between begin/end.

3. Create Entities and Add Components

m_world.create() returns an Entity handle. Use add_component to attach data to it. The entity's archetype is determined by its set of components.

auto e = m_world.create();
m_world.add_component(e, Transform{{ 0, 0, 0 }});
m_world.add_component(e, Sprite{{ 1, 1, 1, 1 }, { 32, 32 }});

4. Query and Update

The query method iterates over all entities that have the specified components. This is the core ECS pattern: systems operate on components, not on specific entities.

m_world.query<Transform, Sprite>([&](kairo::Entity, Transform& t, Sprite&) {
    t.position.x += 50.0f * dt;
});

5. Render

All draw calls go between Renderer::begin(camera) and Renderer::end(). The renderer batches and sorts everything for you.

6. Bootstrap and Run

In main(), create an Engine, initialize it with a window config, and call run() with your application. The engine manages the game loop, timing, and shutdown.

Project Structure

The engine source is organized into modules under engine/:

engine/
  core/         // types, platform, logging, time
  math/         // vectors, matrices, SIMD
  ecs/          // world, archetypes, entities, queries
  graphics/     // renderer, shaders, textures, lights, shadows, post-processing
  input/        // keyboard, mouse, gamepad
  scene/        // scene stack (push/pop/switch), JSON serialization
  assets/       // asset loading and caching
  physics/      // colliders, rigidbodies, spatial grid, raycasting, joints
  particles/    // emitters, particle pools
  events/       // event bus, typed events
  gameplay/     // tilemap, pathfinding, steering, animation
  editor/       // ImGui panels, gizmos, undo system
  scripting/    // Lua bindings, hot-reload
  audio/        // miniaudio wrapper, spatial audio
  memory/       // pool, linear, stack allocators

The included demo game lives in sandbox/. Third-party dependencies are in external/.

Creating a New Game Target

To create your own game using Kairo:

1. Create a Directory

Add a new directory alongside sandbox/. For example, my_game/.

2. Add a CMakeLists.txt

Create my_game/CMakeLists.txt that defines your executable and links against the engine:

// my_game/CMakeLists.txt
// add_executable(my_game main.cpp)
// target_link_libraries(my_game PRIVATE kairo_engine)
// target_include_directories(my_game PRIVATE ${CMAKE_SOURCE_DIR}/engine)

In CMake syntax:

add_executable(my_game main.cpp)
target_link_libraries(my_game PRIVATE kairo_engine)
target_include_directories(my_game PRIVATE ${CMAKE_SOURCE_DIR}/engine)

3. Register It in the Root CMakeLists

Add add_subdirectory(my_game) to the root CMakeLists.txt.

4. Write Your Game

Create my_game/main.cpp following the Quick Start pattern: subclass Application, override the lifecycle methods, and set up your ECS world.

5. Build and Run

cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)
./build/bin/my_game

Key Concepts

Entity Component System (ECS)

Kairo uses an archetype-based ECS. This is the core architectural pattern of the engine.

  • Entities are lightweight handles (64-bit IDs). They have no data or behavior on their own.
  • Components are plain structs that hold data. An entity's set of components (its "archetype") determines how it is stored in memory.
  • Systems are functions (or classes) that operate on entities by querying for specific component combinations.

The archetype approach stores entities with the same component set in contiguous arrays. This means iterating over <Transform, Sprite> touches only packed memory with no indirection, which is extremely cache-friendly.

See the ECS section in the API reference for full details.

Scenes

Scenes are a higher-level organizational unit. Each Scene owns its own World (ECS instance) and Camera. The SceneManager maintains a stack of scenes:

  • Push a scene to overlay it (e.g., a pause menu on top of gameplay)
  • Pop to return to the previous scene
  • Switch to replace the entire stack (e.g., going from menu to gameplay)

Enable scene management by setting m_use_scenes = true in your Application subclass. See the Scenes section for the full API.

The Game Loop

Kairo's engine loop runs as follows each frame:

  1. Input polling -- keyboard, mouse, and gamepad state is updated
  2. Fixed update (on_fixed_update) -- runs at a fixed 60 Hz timestep, accumulating time between frames. Use for physics and deterministic logic.
  3. Variable update (on_update) -- runs once per frame with the actual delta time. Use for gameplay, animation, and camera updates.
  4. Render (on_render) -- draw everything. The renderer batches draw calls between begin/end.
  5. Editor UI (on_editor_ui) -- ImGui panels, if the editor is toggled on with F1.
  6. Swap buffers -- present the frame.

The fixed timestep ensures physics behaves consistently regardless of framerate. The variable timestep gives you smooth rendering and responsive input.
