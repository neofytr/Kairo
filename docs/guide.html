Getting Started

Build the engine from source and create your first Kairo game in under five minutes.

Building from Source #

Requirements

  • C++17 compiler (GCC 9+ or Clang 10+)
  • CMake 3.20+
  • GLFW 3.3+ (install via your package manager)

Everything else is bundled in external/: GLAD, stb_image, ImGui, Lua 5.4, miniaudio, nlohmann/json.

Build Steps

1

Install GLFW:

bash 
# Ubuntu / Debian
sudo apt install libglfw3-dev

# Fedora
sudo dnf install glfw-devel

# Arch
sudo pacman -S glfw-x11
2

Build the engine and demo:

bash 
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)
3

Run the demo:

bash 
./build/bin/sandbox
4

Run tests (263 of them):

bash 
./build/bin/tests

Developed on Linux. Should be reasonably portable to other platforms with OpenGL 4.6 support, but Windows and macOS builds have not been tested.

Your First Game #

Here is the smallest possible Kairo game: a white sprite that moves across the screen.

Step 1 — Include the essentials

You need the engine core, the renderer, a camera, and the ECS world:

C++ 
#include "core/engine.h"
#include "core/application.h"
#include "graphics/renderer.h"
#include "graphics/camera.h"
#include "ecs/world.h"

Step 2 — Define your components

Components are plain C++ structs. No macros, no registration. Just data.

C++ 
struct Transform { kairo::Vec3 position; float rotation = 0.0f; };
struct Sprite    { kairo::Vec4 color; kairo::Vec2 size; };

Step 3 — Create the Application subclass

Override on_init to set up your camera and spawn an entity. Override on_update for gameplay logic and on_render for drawing.

C++ 
class MyGame : public kairo::Application {
    kairo::World m_world;
    kairo::Camera m_camera;

    void on_init() override {
        m_camera.set_orthographic(1280.0f, 720.0f);
        auto e = m_world.create();
        m_world.add_component(e, Transform{{ 0, 0, 0 }});
        m_world.add_component(e, Sprite{{ 1, 1, 1, 1 }, { 32, 32 }});
    }

    void on_update(float dt) override {
        m_world.query<Transform, Sprite>([&](kairo::Entity, Transform& t, Sprite&) {
            t.position.x += 50.0f * dt;
        });
    }

    void on_render() override {
        kairo::Renderer::begin(m_camera);
        m_world.query<Transform, Sprite>([](kairo::Entity, Transform& t, Sprite& s) {
            kairo::Renderer::draw_quad(t.position, s.size, t.rotation, s.color);
        });
        kairo::Renderer::end();
    }
};

Step 4 — Write main()

Initialize the engine, run your game, and shut down when the window closes.

C++ 
int main() {
    kairo::Engine engine;
    if (!engine.init({ { "My Game", 1280, 720, true } })) return 1;
    MyGame game;
    engine.run(game);
    engine.shutdown();
}

That is it. You have an ECS world, a camera, an entity with two components, and a rendering loop. Everything else in the engine — physics, audio, particles, scripting — layers on top of this foundation.

Project Structure #

The engine is split into 16 modules with minimal cross-dependencies:

text 
engine/
  core/         types, platform, logging, time
  math/         vectors, matrices, SIMD
  ecs/          world, archetypes, entities, queries
  graphics/     renderer, shaders, textures, lights, shadows, post-processing
  input/        keyboard, mouse, gamepad
  scene/        scene stack (push/pop/switch), JSON serialization
  assets/       asset loading and caching
  physics/      colliders, rigidbodies, spatial grid, raycasting, joints
  particles/    emitters, particle pools
  events/       event bus, typed events
  gameplay/     tilemap, pathfinding, steering, animation
  editor/       ImGui panels, gizmos, undo system
  scripting/    Lua bindings, hot-reload
  audio/        miniaudio wrapper, spatial audio
  memory/       pool, linear, stack allocators

Creating a Game Target #

The demo lives in sandbox/. To create your own game target:

1

Create a new directory alongside sandbox/:

bash 
mkdir mygame
touch mygame/main.cpp
touch mygame/CMakeLists.txt
2

Write a minimal CMakeLists.txt:

cmake 
add_executable(mygame main.cpp)
target_link_libraries(mygame PRIVATE kairo_engine)
3

Add it to the root CMakeLists.txt:

cmake 
add_subdirectory(mygame)

Then rebuild. Your new executable will appear in build/bin/mygame.

Key Concepts #

Archetype ECS

Entities are 64-bit handles: 32 bits index, 32 bits generation. The generation counter catches use-after-destroy bugs at runtime. Components are stored in packed arrays grouped by archetype (the unique set of component types an entity has). Queries iterate only contiguous memory — no indirection, no cache misses.

Batch Rendering

Sprites are sorted by layer and z-depth, then batched with up to 16 textures per draw call using texture arrays. The common case of many sprites with different textures does not turn into one draw call per sprite.

Fixed vs. Variable Timestep

Use on_fixed_update(dt) for physics (runs at 60 Hz by default). Use on_update(dt) for gameplay and rendering with variable frame time. This separation keeps physics deterministic regardless of frame rate.

Design Decisions

  • Archetype ECS over sparse sets — iteration performance wins over structural change speed. The workload is "iterate all entities with X and Y" far more than "add/remove components on one entity."
  • Custom physics over Box2D — tighter fit for top-down games. No unused features, collision layers that map to game concepts, broadphase tuned for actual entity counts.
  • OpenGL 4.6 over Vulkan — DSA cleans up the API, batch rendering keeps draw calls low, and development time stays reasonable for a 2D engine.
